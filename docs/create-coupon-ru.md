# Create Coupon: инструкция для админки

Документ описывает экран создания купона в Filament admin: `Promotion -> Coupons -> Create`.

Купон состоит из нескольких частей:

- **Main** - базовые настройки купона: код, название, статус, срок действия.
- **Usage Conditions** - условия, при которых купон можно применить в корзине.
- **Issue Conditions** - условия, при которых купон можно выдать клиенту.
- **Actions** - что купон дает клиенту: скидку, бесплатную доставку, товар или combo.
- **Compatibility** - можно ли совмещать купон с BellCoin и другими купонами.
- **Limits** - общие лимиты использования купона.

## Общий порядок создания

1. Откройте `Promotion -> Coupons`.
2. Нажмите `Create`.
3. Заполните вкладку **Main**.
4. Добавьте условия во вкладке **Usage Conditions**, если купон не должен работать для всех заказов.
5. Добавьте **Actions** - без action купон ничего не даст клиенту.
6. Настройте **Compatibility** и **Limits**.
7. Сначала сохраните купон в статусе `draft`.
8. Проверьте купон через симуляцию или тестовый checkout.
9. После проверки переведите статус в `active`.

## Вкладка Main

Здесь задается основная информация о купоне.

### `code`

Промокод, который клиент вводит в приложении или на сайте.

Примеры:

- `WELCOME20`
- `FREEDELIVERY`
- `COMBOGIFT`

Поле обязательное. Любой купон или промокод должен иметь уникальный `code`.

Важно: клиент всегда передает именно код купона. Отдельный id купона клиента в checkout не используется.

### `name`

Внутреннее название купона для админки.

Примеры:

- `Скидка 20% для новых клиентов`
- `Бесплатная доставка от 100 000`
- `Подарочный combo для VIP`

Это обязательное поле. Название должно быть понятным для менеджера, чтобы по нему было ясно, для какой акции создан купон.

### `coupon_kind`

Тип купона.

Доступные значения:

- `public_code_coupon` - публичный промокод.
- `issued_customer_coupon` - купон, который может быть создан под отдельную кампанию или клиента, но все равно применяется только через уникальный `code`.

В текущем checkout оба типа ищутся по `code`. Если нет отдельной логики выдачи, используйте `public_code_coupon`.

### `status`

Статус купона.

Доступные значения:

- `draft` - черновик. Купон сохранен, но не должен применяться в checkout.
- `active` - активен. Купон может применяться, если прошли все условия.
- `paused` - временно остановлен.
- `expired` - истек.
- `archived` - архивный купон, который больше не используется.

Рекомендуемый процесс: сначала создать в `draft`, проверить, потом перевести в `active`.

### `starts_at`

Дата и время начала действия купона.

Если поле пустое, купон не имеет ограничения по началу и может работать сразу после перевода в `active`.

### `ends_at`

Дата и время окончания действия купона.

Если поле пустое, купон не имеет автоматической даты окончания. Для временных акций лучше всегда заполнять `ends_at`.

### `priority`

Приоритет купона. Чем выше значение, тем важнее купон при выборе между несколькими подходящими предложениями.

Если специальных правил нет, оставьте `0`.

### `auto_apply`

Автоматически применять купон без ручного ввода кода.

Используйте осторожно. Обычно это нужно для системных акций, которые должны применяться сами, когда условия выполнены.

### `visible_to_customer`

Показывать купон клиенту.

- Включено - купон можно показывать в клиентском интерфейсе.
- Выключено - купон скрыт от клиента, но может работать как внутренняя/system акция.

### `requires_code_input`

Требовать ввод кода.

- Включено - клиент должен ввести промокод.
- Выключено - купон может применяться автоматической логикой, но checkout все равно должен получить `coupon_code`, если применяет конкретный купон.

Для обычного промокода обычно включено.

### `stackable`

Общий флаг совместимости купона с другими incentive-механиками.

Если выключено, купон лучше рассматривать как отдельную акцию, которую нельзя свободно комбинировать с другими скидками.

## Вкладка Usage Conditions

Здесь задаются условия применения купона в заказе.

Если условий нет, купон считается применимым ко всем заказам, но только при условии что он `active`, попадает в даты действия и не превышены лимиты.

Каждое условие добавляется отдельной строкой в repeater.

### `condition_type`

Тип условия.

Доступные значения:

- `cart.min_amount` - минимальная сумма заказа.
- `customer.segment` - клиент должен быть в нужном сегменте.
- `cart.has_combo` - в корзине должен быть combo.

### `operator`

Оператор сравнения.

Сейчас в форме доступны:

- `>=`
- `=`

Для `cart.min_amount` обычно используйте `>=`.

### `value`

Значение условия.

Примеры:

- Для `cart.min_amount`: `100000`
- Для `customer.segment`: `vip`

Для `cart.has_combo` значение обычно не нужно: система просто проверяет, есть ли в корзине товар с `comboIikoId`.

### `payload`

Дополнительные параметры условия в формате key-value.

Для `customer.segment` можно использовать:

```text
segment_code = vip
```

Если `segment_code` указан в `payload`, система использует его. Если нет - берет значение из поля `value`.

## Вкладка Issue Conditions

Эта вкладка нужна для условий выдачи купона клиенту.

Отличие от **Usage Conditions**:

- **Usage Conditions** проверяются в момент применения купона в checkout.
- **Issue Conditions** должны описывать, кому и когда купон можно выдать.

Форма условий такая же:

- `condition_type`
- `operator`
- `value`
- `payload`

Для гарантии доставки используется:

```text
condition_type = order.delivery_time_over
operator = >
value = 35
```

Это означает: если Kafka event `Delivered` содержит `timestamps.cooking_start_at` и `timestamps.delivered_at`, и разница больше 35 минут, система может автоматически создать клиенту уникальный купон.

Обычный публичный промокод обычно не требует **Issue Conditions**.

Гарантийный template coupon нужно создавать так:

```text
code = DELIVERY_GUARANTEE
coupon_kind = issued_customer_coupon
status = active
```

Клиент не использует `DELIVERY_GUARANTEE` напрямую. Система создает новый уникальный код на основе template coupon, например:

```text
DELIVERY-GUARANTEE-6918295-A1B2C3
```

## Вкладка Actions

Это самая важная вкладка. Здесь задается, что купон делает.

Если action не добавлен, купон может пройти условия, но не даст клиенту скидку или подарок.

### `action_type`

Тип действия.

Доступные значения:

- `percent` - процентная скидка.
- `fixed` - фиксированная скидка.
- `free_delivery` - бесплатная доставка.
- `free_product` - бесплатный товар.
- `free_combo` - бесплатное combo.

### `value`

Размер скидки.

Используется только для:

- `percent`
- `fixed`

Примеры:

- `percent`, `value = 20` означает скидку 20%.
- `fixed`, `value = 25000` означает скидку 25 000 в минимальной денежной единице проекта.

### `max_discount_amount`

Максимальный размер скидки для `percent`.

Пример:

```text
action_type = percent
value = 20
max_discount_amount = 30000
```

Это означает: скидка 20%, но не больше 30 000.

### `product_iiko_id`

iiko product id для `free_product`.

Заполняется только если `action_type = free_product`.

Значение нужно брать из синхронизированных iiko products.

### `combo_iiko_id`

iiko combo id для `free_combo`.

Заполняется только если `action_type = free_combo`.

Значение нужно брать из синхронизированных iiko combos.

### `quantity`

Количество бесплатных товаров или combo.

Используется для:

- `free_product`
- `free_combo`

Если не уверены, ставьте `1`.

### `price_override`

Цена, которую нужно передать для бесплатного товара или combo.

Обычно для подарка ставится `0`.

## Вкладка Compatibility

Здесь настраивается совместимость купона с другими механиками.

### `Can be used with BellCoin`

Можно ли использовать купон вместе со списанием BellCoin.

- Включено - клиент может получить эффект купона и дополнительно использовать BellCoin.
- Выключено - купон не должен совмещаться с BellCoin.

### `Can be used with other coupons`

Можно ли использовать этот купон вместе с другими купонами.

- Включено - купон можно комбинировать с другими купонами.
- Выключено - купон должен применяться отдельно.

Для большинства промо безопаснее оставить выключенным, чтобы не получить слишком большую суммарную скидку.

## Вкладка Limits

Здесь задаются два разных типа лимитов:

- **Usage limits** - сколько раз уже созданный купон можно применить в checkout.
- **Issue limits** - сколько раз template coupon может автоматически создать новые уникальные купоны.

Простое правило:

```text
usage_* = ограничивает применение кода
issue_* = ограничивает автоматическую выдачу новых кодов
```

## Наглядная схема

### Обычный промокод

Один код используют много клиентов:

```text
code = PIZZA20
```

Нужны в основном:

```text
usage_limit_total
usage_limit_per_customer
```

`issue_limit_*` обычно пустые.

### Гарантийный template coupon

Template не применяется клиентом напрямую:

```text
code = DELIVERY_GUARANTEE
coupon_kind = issued_customer_coupon
```

Когда приходит Kafka event `Delivered` и доставка была дольше 35 минут, система создает новый уникальный код:

```text
DELIVERY-GUARANTEE-6918295-A1B2C3
```

Для template важны:

```text
issue_limit_total
issue_limit_per_customer
```

А для каждого созданного уникального купона система автоматически ставит:

```text
usage_limit_total = 1
usage_limit_per_customer = 1
```

То есть гарантийный купон можно использовать один раз.

## Быстрая таблица

| Поле | Что ограничивает | Где работает | Когда заполнять |
| --- | --- | --- | --- |
| `usage_limit_total` | Общее количество применений конкретного кода | checkout apply | Для массовых промокодов и одноразовых кодов |
| `usage_limit_per_customer` | Сколько раз один клиент может применить этот код | checkout apply | Почти всегда ставить `1`, если купон одноразовый |
| `issue_limit_total` | Сколько уникальных кодов можно создать из template | automatic issuing | Для гарантийных/автоматических купонов |
| `issue_limit_per_customer` | Сколько уникальных кодов можно выдать одному клиенту | automatic issuing | Для защиты от повторных компенсаций |
| `issue_policy` | Дополнительные правила выдачи | automatic issuing | Только если backend уже поддерживает это правило |

### `usage_limit_total`

Общий лимит применений купона всеми клиентами.

Пример:

```text
code = PIZZA20
usage_limit_total = 1000
```

Это означает: `PIZZA20` можно применить максимум 1000 раз всеми клиентами вместе.

Если поле пустое, общего лимита нет.

Важно: лимит считает применения в статусах `reserved` и `applied`. Если клиент начал checkout и купон зарезервировался, лимит временно занят. Если заказ не завершился, reservation позже отменяется TTL job.

### `usage_limit_per_customer`

Лимит применений на одного клиента.

Пример:

```text
code = PIZZA20
usage_limit_per_customer = 1
```

Это означает: один клиент может использовать `PIZZA20` только один раз.

Если поставить:

```text
usage_limit_per_customer = 3
```

то один клиент сможет применить этот код до 3 раз.

### `issue_limit_total`

Общий лимит автоматической выдачи новых уникальных кодов из template coupon.

Пример для гарантии доставки:

```text
template code = DELIVERY_GUARANTEE
issue_limit_total = 500
```

Это означает: система может создать максимум 500 гарантийных кодов из template `DELIVERY_GUARANTEE`.

Когда лимит достигнут, новые гарантийные купоны больше не создаются, даже если доставка была дольше 35 минут.

Для обычного промокода это поле обычно оставляем пустым.

### `issue_limit_per_customer`

Лимит автоматической выдачи на одного клиента.

Пример:

```text
template code = DELIVERY_GUARANTEE
issue_limit_per_customer = 1
```

Это означает: один клиент может получить гарантийный купон только один раз по этому template.

Если в Kafka payload нет `customer.id`, система использует `customer.phone` как идентификатор клиента для этого лимита.

Пример:

```text
customer.phone = +998332002852
```

Если этот телефон уже получил гарантийный купон из template `DELIVERY_GUARANTEE`, новый код не создастся.

### `issue_policy`

Дополнительная политика выдачи в формате key-value.

Сейчас для delivery guarantee основная логика берется из **Issue Conditions**:

```text
condition_type = order.delivery_time_over
operator = >
value = 35
```

`issue_policy` можно оставить пустым.

Используйте `issue_policy` только если есть конкретное правило, которое поддерживает backend. Если правила нет, оставьте поле пустым.

## Готовые настройки лимитов

### Массовый промокод на 1000 использований

```text
usage_limit_total = 1000
usage_limit_per_customer = 1
issue_limit_total =
issue_limit_per_customer =
```

### Одноразовый уникальный купон

```text
usage_limit_total = 1
usage_limit_per_customer = 1
issue_limit_total =
issue_limit_per_customer =
```

### Template для гарантийной доставки

```text
usage_limit_total =
usage_limit_per_customer =
issue_limit_total = 500
issue_limit_per_customer = 1
```

Новые коды, созданные из этого template, получат:

```text
usage_limit_total = 1
usage_limit_per_customer = 1
```

## Чеклист перед переводом в active

Проверьте:

- У купона заполнено понятное `name`.
- Заполнен уникальный `code`.
- `status` пока `draft`, если купон еще не проверен.
- `starts_at` и `ends_at` соответствуют периоду акции.
- Во вкладке **Actions** есть хотя бы одно действие.
- Для `percent` заполнен `value`, а при необходимости `max_discount_amount`.
- Для `fixed` заполнен `value`.
- Для `free_product` заполнен существующий `product_iiko_id`.
- Для `free_combo` заполнен существующий `combo_iiko_id`.
- Лимит `usage_limit_per_customer` задан, если купон нельзя использовать повторно.
- Совместимость с BellCoin и другими купонами настроена осознанно.
- Купон проверен через тестовый checkout или симуляцию.

После этого можно менять `status` на `active`.

## Важные замечания

- `active` недостаточно сам по себе: купон также должен попадать в даты действия и лимиты.
- Если `ends_at` пустой, купон не закончится автоматически.
- Если нет **Usage Conditions**, купон может применяться к любому заказу.
- Если нет **Actions**, купон не даст клиенту выгоду.
- Checkout применяет купон только по `coupon_code`.
